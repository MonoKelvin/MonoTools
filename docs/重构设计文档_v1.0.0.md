## 1. 项目现状分析

### 1.1 当前项目定位

当前 MonoTools 是一个**功能丰富的插件平台**，包含以下核心能力：

- ✅ 应用启动器（核心功能）
- ✅ 插件系统（UI插件/无界面插件）
- ✅ 剪贴板管理
- ✅ 超级面板
- ✅ 分离窗口
- ❌ MCP Server（计划删除）
- ❌ 提供商系统（翻译/OCR，计划删除）
- ❌ WebDAV 同步（计划删除）
- ❌ 离线翻译（计划删除）
- ❌ HTTP API 服务（计划删除）
- ❌ 日志收集器（计划删除）
- ❌ 开发者工具（计划删除）
- ❌ ZBrowser 浏览器自动化（计划删除）
- ❌ AI 对话集成（计划删除）

## 2. 重构目标

### 2.1 主要目标

1. **精简项目**: 删除不必要功能，定位为**轻量级系统启动器**
2. **名称统一**: 将所有 ZTools 相关名称替换为 MonoTools
3. **插件简化**: 插件作为内置模块加载，支持热插拔
4. **代码质量**: 移除技术债务，提高可维护性
5. **版本重置**: 从 2.6.1 → 1.0.0（全新起点）

## 5. 插件系统改造

### 5.1 当前架构问题

#### 问题 1: 外部 CLI 依赖

**现状**: 使用 `ztools-plugin-cli` 子模块创建和发布插件

**问题**:
- 依赖外部 Git 子模块
- 需要单独维护和发布
- 增加项目复杂度

**解决方案**: **将 CLI 功能内联到主项目**

#### 问题 2: 插件开发服务器独立运行

**现状**: `internalPluginServer.ts` 启动独立 HTTP 服务器

**问题**:
- 增加资源消耗
- 配置复杂
- 开发体验不佳

**解决方案**: **简化开发模式，直接加载本地文件**

### 5.2 插件系统改造方案

#### 方案 A: 内联插件 CLI（推荐）

**目标**: 移除 `ztools-plugin-cli`，在设置插件中提供插件创建功能

**实现步骤**:

1. **创建插件模板引擎**

```typescript
// src/main/core/pluginTemplates/pluginTemplateEngine.ts

export class PluginTemplateEngine {
  /**
   * 生成插件项目文件结构
   */
  async createPlugin(projectName: string, options: PluginOptions): Promise<string> {
    const pluginDir = path.join(app.getPath('userData'), 'plugin-templates', projectName)

    // 生成 plugin.json
    await this.generatePluginJson(pluginDir, options)

    // 生成 preload.js
    await this.generatePreload(pluginDir, options)

    // 生成 package.json
    await this.generatePackageJson(pluginDir, options)

    // 生成 README.md
    await this.generateReadme(pluginDir, options)

    return pluginDir
  }

  private async generatePluginJson(dir: string, options: PluginOptions): Promise<void> {
    const pluginJson = {
      name: options.name,
      version: '1.0.0',
      description: options.description,
      main: options.hasUI ? 'index.html' : undefined,
      features: []
    }
    await fs.writeJson(path.join(dir, 'plugin.json'), pluginJson, { spaces: 2 })
  }

  // ... 其他生成方法
}
```

2. **在设置插件中添加"创建插件"功能**

```vue
<!-- internal-plugins/setting/src/views/PluginMarketSetting/PluginMarketSetting.vue -->
<template>
  <div>
    <FeatureCard @click="createNewPlugin" title="创建新插件" icon="plus" />
    <!-- 其他功能 -->
  </div>
</template>

<script setup lang="ts">
async function createNewPlugin() {
  // 调用主进程 API

    name: 'my-plugin',

</script>
```

3. **添加插件发布功能**

```typescript
// src/main/api/plugin/internal.ts

class PluginInternalAPI {
  /**
   * 创建新插件项目
   */
  public async createPlugin(options: CreatePluginOptions): Promise<string> {
    const engine = new PluginTemplateEngine()
    const pluginDir = await engine.createPlugin(options.name, options)
    return pluginDir
  }

  /**
   * 打包插件为 ZPX 格式
   */
  public async packagePlugin(pluginDir: string): Promise<string> {
    const zpxPath = await this.packageToZpx(pluginDir)
    return zpxPath
  }
}
```

## 9. 风险评估

### 9.1 高风险

#### 🔴 插件兼容性破坏

**风险**: 删除的 API 可能被现有插件使用

**影响**: 插件无法正常运行

**缓解措施**:
1. **API 兼容层**: 在删除前先标记为 deprecated，保留一个版本周期
2. **文档通知**: 提前在文档和更新日志中说明
3. **版本号跳升**: 从 2.x 直接跳到 1.0.0，明确表示不兼容

```typescript
// 在 2.7.0 版本添加废弃警告
console.warn('[Deprecation] MCP API will be removed in v1.0.0')
```

#### 🔴 数据迁移失败

**风险**: 现有用户数据迁移失败

**影响**: 用户丢失插件数据、历史记录等

**缓解措施**:
1. **保留迁移工具**: 不删除 `databaseMigration.ts`，改为手动工具
2. **数据备份**: 在设置中添加"导出数据"功能
3. **迁移日志**: 详细记录迁移过程

### 9.2 中风险

#### 🟡 功能遗漏

**风险**: 删除功能时遗漏相关代码

**影响**: 应用崩溃或出现错误

**缓解措施**:
1. **清单检查**: 严格按文档中的删除清单执行
2. **代码搜索**: 删除后全局搜索相关关键词
3. **测试覆盖**: 充分测试所有功能

#### 🟡 名称替换遗漏

**风险**: ZTools → MonoTools 替换不完整

**影响**: 用户困惑，专业度下降

**缓解措施**:
1. **全局搜索**: 删除完成后全局搜索 "ZTools"、"ZTOOLS"
2. **文档审查**: 逐行检查文档
3. **UI 审查**: 检查所有设置页面和提示信息

### 9.3 低风险

#### 🟢 性能回归

**风险**: 重构后性能下降

**缓解措施**:
1. 性能基准测试
2. 代码审查

#### 🟢 开发效率下降

**风险**: 删除功能后开发效率降低

**缓解措施**:
1. 保留核心 API
